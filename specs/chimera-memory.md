# Chimera Memory — Architecture Spec v0.1

**Status:** Draft
**Author:** Tiger + Clawd
**Date:** 2026-04-07

## Vision

Combine the best of three MIT-licensed memory systems into one:
- **MemPalace** — verbatim storage, palace spatial architecture, AAAK compression
- **Hindsight** — multi-strategy retrieval (TEMPR), entity resolution, typed memory
- **Hermes** — MemoryProvider plugin interface, auto-injection, lifecycle hooks

Goal: highest retrieval accuracy, zero information loss, fully local, no server needed.

---

## Core Principles

1. **Store everything verbatim** — no summarization, no extraction, no "AI decides what matters" (MemPalace philosophy)
2. **Structure before search** — palace architecture pre-filters by domain before retrieval fires (MemPalace)
3. **Multi-strategy retrieval** — semantic + keyword + entity graph + temporal (Hindsight TEMPR)
4. **Relevance threshold** — if nothing scores above threshold, inject NOTHING (novel — none of the 3 do this)
5. **Fully local** — ChromaDB + SQLite, no PostgreSQL, no Docker, no server, no cloud

---

## Architecture

### Layer 1: Storage (from MemPalace)

**Backend:** ChromaDB (PersistentClient) + SQLite

**What gets stored:** Verbatim text chunks. No summarization. Original words preserved.

**Palace structure via metadata fields on ChromaDB documents:**
- `wing` — domain/person/project (e.g., "fiona", "basicfit", "trading", "health")
- `hall` — memory type (facts, experiences, decisions, preferences, emotional)
- `room` — specific topic within the wing (e.g., "interview", "contract-law", "karolina")
- `source` — where it came from (conversation, file, manual)
- `timestamp` — when it happened (not when stored)
- `speakers` — who said what (for conversation data)

**Chunk strategy:** Conversation exchanges (Q+A pairs) as atomic units, not arbitrary char splits.

**Truth tiers — every drawer gets a `truth_tier` metadata field:**
- `evidence` — verbatim text from conversations, things that actually happened. Ground truth. DEFAULT for sync_turn and conversation imports. Example: "Yasin said 'expect my call Friday 10:00'"
- `inference` — patterns/connections generated by LLM analysis during mining. Flagged automatically, displayed with lower confidence. Example: "Tiger tends to self-sabotage at peak success moments"
- `canon` — user-confirmed facts, explicitly validated or stored via chimera_store. Highest confidence, never auto-pruned. Example: "Phil, 34, Bremen, BasicFit"

**Why this matters:**
- Model degradation (like Opus recently) generates bad inferences. Without tier separation, bad inferences get stored with same authority as direct quotes. One bad session corrupts memory permanently.
- With tiers: evidence is immutable ground truth, inferences can be pruned/questioned, canon is user-validated.
- Phase 1 menu shows truth tier so agent can prioritize evidence+canon over inference.

**Phase 1 display with truth tiers:**
```
Palace scan:
› basicfit/interview [EVIDENCE] (89%) — R2 answers, Yasin feedback, Friday callback
› personality/patterns [INFERENCE] (72%) — self-sabotage at peaks, protective system  
› identity/core [CANON] (95%) — Phil, 34, Bremen, BasicFit host
```

**Drawer IDs:** `{wing}_{room}_{content_hash[:16]}`

### Layer 2: Knowledge Graph (hybrid MemPalace + Hindsight)

**Backend:** SQLite (like MemPalace's knowledge_graph.sqlite3)

**Tables:**
- `entities` — canonical names, first_seen, last_seen, mention_count
- `entity_mentions` — entity_id → drawer_id (many-to-many)
- `entity_cooccurrences` — which entities appear together, count
- `triples` — subject → predicate → object with timestamps (MemPalace KG)
- `temporal_index` — drawer_id → event_date range for temporal queries

**Entity resolution:** Trigram fuzzy matching + co-occurrence scoring (from Hindsight).
Name variants map to canonical: "Phil" = "Tiger" = "Phil N" → entity_id:1

### Layer 3: Retrieval — TEMPR Lite (from Hindsight, simplified)

Four parallel retrieval strategies, NO PostgreSQL needed:

**A) Semantic (ChromaDB native)**
- all-MiniLM-L6-v2 embeddings (ChromaDB default)
- Palace pre-filter: `where={"wing": detected_wing}` narrows search before embedding comparison
- Over-fetch 3x, trim in Python

**B) BM25 Keyword (SQLite FTS5)**
- SQLite FTS5 virtual table indexing all drawer text
- Catches exact keyword matches that semantic search misses
- "What did Karolina say about the schedule?" → BM25 finds "Karolina" + "schedule" even if embeddings drift

**C) Entity Graph (SQLite joins)**
- From semantic seeds, expand through entity_mentions table
- "Find everything about Fiona" → entity_id → all drawers mentioning Fiona → related entities (Tiger, BasicFit, etc.)
- Uses co-occurrence to find connected topics

**D) Temporal (SQLite date range)**
- Extract date intent from query: "last month", "in January", "yesterday"
- Filter temporal_index by date range
- Then semantic similarity within the time-filtered set

**Rank Fusion:** Reciprocal Rank Fusion (k=60) merging all 4 lists (from Hindsight).

**Reranking:** MiniMax as cross-encoder reranker (free for us, Sonnet-level quality).
- Score = RRF_base × recency_boost × temporal_proximity_boost
- Recency: linear decay over 365 days (recent memories score slightly higher)

**CRITICAL: Relevance Threshold**
- If top result score < threshold → inject NOTHING
- This is the single biggest improvement over all three systems
- ByteRover injects 5 random topics every single message. MemPalace/Hindsight have no threshold either
- Configurable: default 0.4, adjustable per-user

### Layer 4: Always-On Context (v1 — no AAAK yet)

**Layer 0 — Entity Registry (~80-120 tokens, ALWAYS loaded every session)**

A structured registry of key people, projects, and facts. Not queried — just always present. The agent knows who everyone is before the first message arrives.

```
SELF: Phil/Tiger | 34 | Bremen | CM candidate BasicFit 5 studios
PEOPLE:
  Fiona — ex, complex history, last contact Jan 2026
  Karolina — coworker BasicFit, nosy, same shifts, 21
  Yasin (Abulqasim Faqraz) — CM hiring manager, R1+R2 interviewer
  Mack — cat, orange, chair dweller, guilt trip specialist
  Dennis — outgoing CM, pointed Tiger to apply
  Asmaiyt — R2 interviewer, Yasin's colleague
PROJECTS:
  Chimera — memory system, this project
  Naturschoen — Shopify shop, self-funding
  Hermes — AI agent framework, contributor
STATUS:
  R2 interview done Apr 7, callback Friday 10:00
  ByteRover disabled, building Chimera replacement
```

**How it's maintained:**
- Built from `canon` tier facts (user-confirmed, highest confidence)
- Updated via on_session_end when new canon facts are established
- Manual override via chimera_store with canon tier
- NOT generated by LLM reasoning (no Honcho-style "story about you")
- Simple structured text, human-editable

**Layer 1 — Top Drawers (~200 tokens, always loaded)**

Most frequently accessed / highest relevance drawers, plain text summaries.
AAAK compression deferred to v2 — plain short summaries for now.

**On-demand layers (Phase 2 — agent double-clicks):**
- Layer 2: Filtered retrieval by wing/room
- Layer 3: Full TEMPR search across palace

### Layer 5: Two-Phase Retrieval (NOVEL — none of the 3 systems do this)

The core innovation: **don't force-feed the agent. Give it a menu.**

Every existing memory system (ByteRover, Hindsight, MemPalace) dumps full content into context.
The agent has no say in what gets injected. This wastes tokens and pollutes reasoning.

Chimera uses two phases:

**PHASE 1 — Palace Overview (auto-injected via prefetch, ~150-200 tokens max)**

Runs automatically before every turn. Returns AAAK-compressed summaries:

```
Palace scan (3 matches above 40% threshold):
› chimera_project/architecture (89%) — TEMPR lite design, relevance threshold, two-phase retrieval
› chimera_project/comparison (72%) — MemPalace vs Hindsight vs ByteRover benchmarks
› basicfit/business (45%) — KB service model, GDPR stack, recurring revenue

Nearby wings: hermes (12 rooms), trading (8 rooms), health (4 rooms)
Use chimera_expand("wing/room") or chimera_search("query") to dive deeper.
```

If NO topic scores above threshold → inject NOTHING. Zero tokens wasted.

**PHASE 2 — Agent Navigates (tool calls, only when agent decides)**

The agent reads the menu and decides:
- `chimera_expand("chimera_project/architecture")` → full verbatim content of that room
- `chimera_search("what did Karolina say about the schedule")` → targeted TEMPR search
- `chimera_graph("fiona")` → entity graph: all rooms/wings mentioning Fiona
- Or: agent decides none are relevant and just answers without memory context

The agent navigates the palace like a building:
- See available wings → zoom into a wing → see rooms → open a room → read drawers
- Only pull full content when actually needed
- Double-click, not force-feed

**Why this is better:**
1. **Token efficiency** — 150 tokens overview vs 4000+ token content dump
2. **Agent judgment** — the LLM understands context better than any similarity score
3. **No garbage** — threshold + agent choice = zero irrelevant injections
4. **Palace metaphor** — spatial navigation is intuitive for any LLM

### Layer 6: Hermes Integration (new MemoryProvider plugin)

**File:** `plugins/memory/chimera/__init__.py`

**Hooks implemented:**

| Hook | What it does |
|------|-------------|
| `initialize()` | Open ChromaDB + SQLite, load AAAK wake-up context |
| `system_prompt_block()` | Return AAAK identity (~120 tokens) + palace map overview |
| `prefetch(query)` | PHASE 1: TEMPR scan → AAAK summaries above threshold, or "" |
| `sync_turn(user, assistant)` | Store exchange verbatim, extract entities, update temporal index |
| `on_pre_compress(messages)` | Flush messages about to be compressed into palace |
| `on_memory_write(action, target, content)` | Mirror builtin memory writes as palace drawers |
| `on_delegation(task, result)` | Store subagent results in palace (NEW — nobody does this) |
| `on_session_end(messages)` | End-of-session entity consolidation + AAAK refresh |
| `get_tool_schemas()` | chimera_expand, chimera_search, chimera_graph, chimera_navigate, chimera_store |
| `handle_tool_call()` | PHASE 2: Agent-driven palace navigation |

**Tool schemas:**

| Tool | Purpose |
|------|---------|
| `chimera_expand(path)` | Open a specific room/wing, get full verbatim content |
| `chimera_search(query)` | Full TEMPR search across entire palace |
| `chimera_graph(entity)` | Entity graph: all connections to a person/project/topic |
| `chimera_navigate(wing)` | List rooms in a wing with drawer counts and freshness |
| `chimera_store(content, wing, room)` | Manually file something in a specific palace location |

**Prefetch flow (Phase 1):**
1. Detect likely wings from query keywords (pre-filter)
2. Run 4 TEMPR strategies in parallel (threads)
3. RRF fusion → ranked candidate list
4. Apply relevance threshold (default 0.4)
5. AAAK-compress top matches into ~150 token overview
6. Include nearby wings hint for navigation
7. If nothing above threshold → return "" (inject nothing)

## Session Import (built-in from day 1)

MemPalace's `normalize.py` already handles:
- Claude JSON exports
- ChatGPT JSON exports  
- Claude Code JSONL (hermes_state.db exports — that's us)
- Slack JSON exports
- Plain text transcripts

**Import pipeline:**
1. Export from hermes_state.db (script already exists from ByteRover work)
2. `chimera mine ~/exports/ --mode convos` → MemPalace convo_miner
3. Auto-detects wings from content (Fiona conversations → wing:fiona)
4. Auto-detects rooms from topics (interview talk → room:interview)
5. Chunks by Q+A exchange pairs (not arbitrary char splits)
6. Entities auto-extracted and stored in SQLite

This is BETTER than ByteRover bulk import because palace structure
organizes on intake — no "everything at importance 50" chaos.

## Migration Path

### From ByteRover:
1. Export context tree markdown files
2. Run chimera mine on exported files (MemPalace miner handles markdown)
3. Entities auto-detected, palace structure auto-generated

### From conversation history:
1. Export from hermes_state.db (we already have the scripts)
2. Run chimera mine --mode convos on exports
3. MemPalace's normalize.py handles Claude/ChatGPT/Slack formats

### From scratch:
1. `chimera init` — set up palace, define wings
2. Hermes auto-populates via sync_turn on every conversation
3. Palace grows organically

## Success Criteria

1. On LoCoMo benchmark: ≥95% (beating ByteRover 92.2% and Hindsight 92%)
2. On LongMemEval: ≥97% with MiniMax rerank
3. Relevance threshold: zero garbage injections (ByteRover currently ~80% garbage)
4. Prefetch latency: <2 seconds (ChromaDB + SQLite, no network calls for retrieval)
5. Storage: verbatim, searchable, human-browsable palace structure
6. Install: `pip install chimera-memory` + zero config for basic use

## Open Questions

1. **Name:** "Chimera" is a working title. Need something memorable for open source.
2. **MiniMax reranker:** Do we call MiniMax API for reranking, or use a local cross-encoder model? Local = faster + free everywhere. MiniMax = better quality but requires API key.
3. **Wing auto-detection:** How aggressively should we auto-create wings from conversation content? Too many wings = fragmented. Too few = no pre-filtering benefit.
4. **AAAK for closets:** MemPalace plans to add AAAK inside closets (compressed drawer summaries). Do we implement this in v1 or defer?
